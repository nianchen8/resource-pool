# 初级定制

> 开箱即用里你用默认配置跑通了 5 种写法。现在的问题是：**怎么拧旋钮，让它更贴合你的场景？**

读完这篇你不再是"抄了就跑"——你会知道每个池有哪些旋钮、每个旋钮叫什么、干什么用、什么时候应该改它。

---

## 旋钮总览：三个池子，26 个旋钮

在你开始改代码之前，先把所有能调的东西看一遍。别怕多——日常真正需要改的也就三四个。

```
UserAgentPool(strategy, thread_safe, data_dir, load_builtin,
              load_fed, raw_only)                       ── 6 个旋钮
DNSResolverPool(strategy, regions, cache_ttl, max_cache_size,
                max_consecutive_fails, revive_after,
                thread_safe, fallback_to_system,
                data_dir, load_builtin, load_fed)        ── 11 个旋钮
ProxyPool(strategy, max_consecutive_fails, revive_after,
          thread_safe, min_alive, auto_refill_url,
          data_dir, load_builtin, load_fed)              ── 9 个旋钮
```

> 注：同步版和异步版本旋钮一致。异步版类名前加 `Async`——`AsyncUserAgentPool`、`AsyncDNSResolverPool`、`AsyncProxyPool`。

## UA 池旋钮详解

### 旋钮 ① `strategy`（选取策略）

```
英文名：strategy
中文名：选取策略
类型：  UAStrategy 枚举
默认值：UAStrategy.WEIGHTED
可选值：UAStrategy.WEIGHTED  |  UAStrategy.UNIFORM
```

**干什么的？** 控制 `pool.get()` 和 `pool.get_headers()` 以什么概率从候选 UA 中挑出一条。

**`WEIGHTED`（加权随机，默认）**

每条种子 UA 在 `ua_seeds.json` 里都带着一个 `weight` 字段（1~10）。`WEIGHTED` 策略下，`weight=10` 的 UA 被选中的概率是 `weight=1` 的 10 倍。内置种子库里 `weight` 较高的多是 Chrome 近期版本——因为这些 UA 最不容易被反爬识别。

选它的场景：你信任内置种子库的权重分配，想让"更像真人"的 UA 多出现。

**`UNIFORM`（均匀随机）**

无视 `weight` 字段，每条候选 UA 命中概率完全相等。不管你加载的是内置种子还是自己的数据，每一条被选中的机会都一样。

选它的场景：你有一套自己积累的、条条都能用的 UA，不想任何一条"被偏爱"。

```python
# 均匀轮换
ua = UserAgentPool(strategy=UAStrategy.UNIFORM)

# 运行时也能改
ua.strategy = UAStrategy.WEIGHTED
```

> ⚠️ 注意：`pool.get()` 和 `pool.get_headers()` 可以传 `weighted=False` 覆盖池级策略——但这是单次覆盖，不影响池级默认值。

---

### 旋钮 ② `thread_safe`（线程安全）

```
英文名：thread_safe
中文名：线程安全
类型：  bool
默认值：True
可选值：True  |  False
```

**干什么的？** 控制池内部是否启用锁。

设为 `True`（默认）：池使用 `ReadWriteLock` 保护内部数据结构。读操作（`get`/`get_headers`）共享读锁，多线程可以同时进入；写操作（`add`/`remove`）持写锁，独占。你不需要在自己的代码里加任何锁。

设为 `False`：所有锁变成空操作（`DummyReadWriteLock`），零开销。

| 什么时候用 `True` | 什么时候用 `False` |
|---|---|
| 多线程脚本（ThreadPoolExecutor） | 单线程脚本 |
| 模块级全局池（多个模块共用） | 多进程子进程内（单线程环境） |
| 不确定该不该关时 | 性能压测极端场景 |

> ⚠️ 同步版的 `thread_safe=True` 锁开销极小（微秒级）。一般情况下不需要关——关错了反而可能数据竞争。

## DNS 池旋钮详解

### 旋钮 ③ `strategy`（服务端选择策略）

```
英文名：strategy
中文名：服务端选择策略
类型：  SelectStrategy 枚举
默认值：SelectStrategy.LATENCY_WEIGHTED
可选值：SelectStrategy.LATENCY_WEIGHTED  |  SelectStrategy.ROUND_ROBIN  |  SelectStrategy.RANDOM
```

**干什么的？** 控制每次 `resolve()` 调用时，从哪台 DNS 服务器开始尝试。

**`LATENCY_WEIGHTED`（延迟加权，默认）**

每台 DNS 服务器记录历史延迟。选择时按"延迟 ÷ 权重"升序排列——延迟低、权重高的优先。第一次使用时所有服务器延迟都是 0，调一次 `health_check()` 后会填充延迟数据。

选它的场景：日常使用——最快的 DNS 多用，慢的少用，保证解析速度。

**`ROUND_ROBIN`（轮询）**

严格轮流使用每台 DNS 服务器。第 1 次请求 → 服务器 A，第 2 次 → 服务器 B……轮完一圈从头开始。

选它的场景：长时间爬取，希望每台 DNS 负载均匀，避免某台被过度使用。

**`RANDOM`（随机）**

每次随机选一台。

选它的场景：你已经做了 `health_check()` 过滤掉不可用服务器，剩下都健康，随便选。

```python
dns = DNSResolverPool(strategy=SelectStrategy.ROUND_ROBIN)

# 运行时也能改
dns.strategy = SelectStrategy.LATENCY_WEIGHTED
```

> ⚠️ `LATENCY_WEIGHTED` 依赖于 `health_check()` 的延迟数据。如果很长时间没跑健康检查，延迟数据会过时，策略退化为"按初始权重排序"。建议生产环境每 5~10 分钟跑一次 `health_check()`。

---

### 旋钮 ④ `regions`（地域线路）

```
英文名：regions
中文名：地域线路
类型：  tuple[str, ...]
默认值：("domestic", "overseas")
可选值：("domestic",)  |  ("overseas",)  |  ("domestic", "overseas")
```

**干什么的？** 控制启用哪些地域的 DNS 服务器。

内置 DNS 服务器阵容：

| 地域 | 数量 | 服务器 |
|---|---|---|
| `domestic`（国内） | 7 台 | 114.114.114.114、223.5.5.5、119.29.29.29 等 |
| `overseas`（海外） | 7 台 | 8.8.8.8、8.8.4.4、1.1.1.1、1.0.0.1 等 |

**什么时候改：**

- 只爬国内站（百度、微博、京东）→ `("domestic",)`——延迟更低，也不会被海外 DNS 返回的境外 CDN 节点绕路
- 只爬海外站（Google、YouTube）→ `("overseas",)`——海外 DNS 解析海外域名更快
- 国内外都爬 → 默认 `("domestic", "overseas")`——池会优先尝试你指定地域的服务器

```python
# 只启用国内 7 台 DNS
dns = DNSResolverPool(regions=("domestic",))
```

> ⚠️ 写 `("domestic")` 不带逗号会被 Python 当成字符串而不是元组——必须是 `("domestic",)` 或 `tuple(["domestic"])`。日志里看到"已加载 7 台 DNS"而不是"14 台"就说明地域过滤生效了。

---

### 旋钮 ⑤ `cache_ttl`（缓存过期时间）

```
英文名：cache_ttl
中文名：缓存过期时间
类型：  int（秒）
默认值：300（5 分钟）
可选值：任意正整数
```

**干什么的？** 控制域名解析结果在内存中缓存多久。

DNS 解析不是每次都去问 DNS 服务器——同一个域名解析过一次后，结果缓存 `cache_ttl` 秒，后续请求直接走缓存，零延迟。

**什么时候改：**

- 目标站 DNS 频繁变动（CDN 切换、负载均衡轮换快）→ 降到 60~120 秒，避免缓存过期 IP
- 爬取速度极快（每秒上千请求）→ 缓存帮大忙，保持默认或调高到 600
- 不在乎缓存、每次都要最新 IP → 设为 1（但不建议——浪费资源还减速）

```python
dns = DNSResolverPool(cache_ttl=120)  # 每 2 分钟刷新一次缓存
```

> 缓存按"域名:记录类型"分 key（如 `"www.baidu.com:A"`）。同一域名不同记录类型各自缓存。

---

### 旋钮 ⑥ `max_cache_size`（最大缓存条目数）

```
英文名：max_cache_size
中文名：最大缓存条目数
类型：  int
默认值：4096
可选值：任意正整数
```

**干什么的？** 控制缓存字典最多存多少条记录。超过上限时，最旧的条目（FIFO）被驱逐。

**什么时候改：**

- 你爬的域名非常多（几千个不同域名）→ 调高，比如 8192 或 16384
- 内存紧张 → 调低，比如 1024

> 一条缓存记录约占用 200~300 字节（IP 列表 + 时间戳）。4096 条 ≈ 1MB 内存。几乎不需要改。

---

### 旋钮 ⑦ `max_consecutive_fails`（连续失败容忍数）

```
英文名：max_consecutive_fails
中文名：连续失败容忍数
类型：  int
默认值：3
可选值：任意正整数
```

**干什么的？** 一台 DNS 服务器连续解析失败多少次后被自动隔离（`enabled=False`）。

比如 `114.114.114.114` 连续 3 次解析失败 → 标记为"不可用" → 后续请求跳过它 → `revive_after` 秒后自动复活（见下一个旋钮）。

**什么时候改：**

- DNS 服务器质量不稳定 → 调到 5（减少误隔离，给它更多机会）
- DNS 服务器很宝贵（自己搭的）→ 调到 2（失败快隔离，防止大量请求浪费在坏服务器上）
- 只有一个 DNS 服务器 → 不用管（隔离后立刻复活，仍然是唯一的候选）

```python
dns = DNSResolverPool(max_consecutive_fails=5)  # 容忍 5 次再隔离
```

---

### 旋钮 ⑧ `revive_after`（隔离复活时间）

```
英文名：revive_after
中文名：隔离复活时间
类型：  int（秒）
默认值：120（2 分钟）
可选值：任意正整数
```

**干什么的？** 被隔离的 DNS 服务器多久后自动重新启用。

隔离不是永久的——隔离时间过了之后，服务器会被重新标记为 `enabled=True`，下次解析时给它"试用"机会。试用成功 → 恢复正常；试用失败 → 重新进入隔离倒计时。

**什么时候改：**

- DNS 服务器是公共的（114、8.8.8.8）→ 保持默认 120 秒。它们偶尔抽风，很快就能恢复
- DNS 服务器经常性抽风（比如某海外线路间歇性丢包）→ 调到 300 或 600，减少"试了又失败"的浪费
- 只有一两台 DNS → 调小到 30 或 60，让它尽快回来

```python
dns = DNSResolverPool(revive_after=300)  # 隔离 5 分钟后才复活
```

---

### 旋钮 ⑨ `thread_safe`（DNS 池版）

和 UA 池的 `thread_safe` 逻辑一样——`True` 启 `threading.Lock`，`False` 变 `DummyLock`。

> DNS 池还用到了**16 路缓存分片锁**（`_CACHE_SHARDS=16`）——按域名首字符哈希到 16 个独立锁。这是内置的，不需要你配置。这意味着 16 个不同域名可以同时读写缓存，不会互相阻塞。

---

### 旋钮 ⑩ `fallback_to_system`（系统 DNS 回退）

```
英文名：fallback_to_system
中文名：系统 DNS 回退
类型：  bool
默认值：True
可选值：True  |  False
```

**干什么的？** 当池内所有 DNS 服务器都解析失败时，是否回退到系统默认 DNS（你电脑网络设置里配的那个）。

设为 `True`：全部失败 → 用系统 DNS 做最后一次尝试。这让你在最坏情况下也能拿到解析结果。

设为 `False`：全部失败 → 直接抛异常，不找系统 DNS。

| 什么时候用 `True`（默认） | 什么时候用 `False` |
|---|---|
| 日常使用——容错最重要 | 你的系统 DNS 被污染了（解析结果不对） |
| 池内至少有一台靠谱服务器 | 你想确保 100% 走池内 DNS（调试用） |

```python
# 拒绝系统 DNS——全部失败就报错
dns = DNSResolverPool(fallback_to_system=False)
```

## Proxy 池旋钮详解

### 旋钮 ⑪ `strategy`（代理选择策略）

```
英文名：strategy
中文名：代理选择策略
类型：  ProxyStrategy(str, Enum) 枚举 | str 字符串
默认值：ProxyStrategy.LATENCY_WEIGHTED（即 "latency_weighted"）
可选值：ProxyStrategy.LATENCY_WEIGHTED  |  ProxyStrategy.ROUND_ROBIN  |  ProxyStrategy.RANDOM
        也支持直接传入字符串："latency_weighted" | "round_robin" | "random"
```

和 DNS 池的三个策略含义完全一样——只是作用对象从 DNS 服务器变成代理 IP。

**`LATENCY_WEIGHTED`（延迟加权）**——延迟最低的代理优先。延迟来自 `health_check()` 的探测结果。如果没跑过健康检查，所有代理延迟为 0，策略退化为按权重排列。

**`ROUND_ROBIN`（轮询）**——严格轮流。适合"所有代理质量差不多，让它们均匀负载"的场景。

**`RANDOM`（随机）**——每次随机选一个。

```python
proxy = ProxyPool(strategy=ProxyStrategy.ROUND_ROBIN)
```

---

### 旋钮 ⑫ `min_alive`（最低存活数）

```
英文名：min_alive
中文名：最低存活数
类型：  int
默认值：0（不检查）
可选值：任意正整数
```

**干什么的？** 设定"至少要有 N 个可用代理"的阈值。配合 `auto_maintain()` 使用——当可用代理数低于 `min_alive` 时，自动从 `auto_refill_url` 拉新代理补充。

**工作流程：**

```
1. 调用 proxy.auto_maintain()
2. 评分淘汰 → 去掉 score < 10 的低分代理
3. 统计存活数 → 如果 < min_alive
4. 自动 load_from_url(auto_refill_url) 补充
```

**什么时候改：**

- 没设 `auto_refill_url` → 保持 0（设了也没用，没地方补充）
- 生产环境，代理供应商 1 元/100 个 → 设 10~20，保持充裕
- 代理很贵 → 设 3~5，够用就行
- 代理按量付费 → 设到刚好覆盖并发数

```python
proxy = ProxyPool(
    min_alive=10,
    auto_refill_url="http://your-provider.com/api?key=xxx",
)
# 每 5 分钟调一次：
# proxy.auto_maintain()   → 存活 < 10 就自动拉
```

> ⚠️ `auto_maintain()` 有 60 秒冷却期——连续调用只有第一次生效。别在每次请求前都调。

---

### 旋钮 ⑬ `auto_refill_url`（自动补充地址）

```
英文名：auto_refill_url
中文名：自动补充地址
类型：  str
默认值：""（空字符串，不启用）
可选值：代理供应商的提取 API URL
```

**干什么的？** 配合 `min_alive` 和 `auto_maintain()` 使用——这是自动补充时代理从哪来的 URL。

支持的 API 返回格式（自动识别）：

| 格式 | 示例 | 常见供应商 |
|---|---|---|
| 纯文本（逐行 `ip:port`） | `183.207.226.9:9999\n120.197.85.171:33965` | 大多数 |
| JSON — IP/Port 分离 | `{"code": 0, "data": [{"IP": "1.2.3.4", "Port": 8888}]}` | 携趣、天启、多米 |
| JSON — proxy_list | `{"data": {"proxy_list": ["1.2.3.4:8080"]}}` | 快代理、91VPS |
| JSON — proxies | `{"proxies": ["1.2.3.4:8080"]}` | 阿布云 |
| JSON — 纯数组 | `["1.2.3.4:8080", "5.6.7.8:3128"]` | 齐云 |

也支持带鉴权的代理格式：`1.2.3.4:8080:user:pass` 或 `1.2.3.4:8080@user:pass`。

```python
proxy = ProxyPool(
    min_alive=10,
    auto_refill_url="http://provider.com/api?key=YOUR_KEY&count=20",
)
# 首次手动拉一批
proxy.load_from_url("http://provider.com/api?key=YOUR_KEY&count=50")
proxy.health_check()  # 先探测一遍延迟
# 之后自动维护
proxy.auto_maintain()  # 淘汰低分 + 不够 min_alive 就补充
```

> ⚠️ 首次使用建议先手动 `load_from_url()` + `health_check()`，确认 API 可用后再开 `auto_maintain()`。否则等到自动维护发现拉不到代理时可能已经晚了。

---

### 旋钮 ⑭ `max_consecutive_fails`、`revive_after`、`thread_safe`（代理池版）

这三个和 DNS 池的同名旋钮**含义完全一样**，作用对象从 DNS 服务器换成代理 IP：

- `max_consecutive_fails`：代理连续失败多少次后被隔离。默认 3。
- `revive_after`：隔离多少秒后自动复活。默认 120。
- `thread_safe`：是否启用锁。默认 `True`。

---

## 编排器旋钮

### `PoolOrchestrator(ua=..., dns=..., proxy=..., **extra_pools)`

```
英文名：PoolOrchestrator / AsyncPoolOrchestrator
中文名：资源编排器
作用：  把 UA、DNS、Proxy（以及你自定义的池）绑在一起，一次调用拿回全套资源
```

**参数：**

- `ua`：`UserAgentPool` 或 `AsyncUserAgentPool` 实例
- `dns`：`DNSResolverPool` 或 `AsyncDNSResolverPool` 实例
- `proxy`：`ProxyPool` 或 `AsyncProxyPool` 实例（可选，没有就不传）
- `**extra_pools`：你的自定义池（深度定制会讲），如 `cookie=my_cookie_pool`

**`orch.next()` 返回什么？** 一个 `PoolCombo` 对象（字典子类），支持属性访问和字典访问两种方式：

```python
c = orch.next()
c.ua        # dict — 完整请求头（14 项）
c.dns       # str  — 本次使用的 DNS 服务器 IP
c.proxy     # dict — {"http": "...", "https": "..."}（如果传了 proxy）

# 字典子类，所以也支持：
c["ua"]     # 等价于 c.ua
{**c}       # 解包
```

## 单线程

和开箱即用的区别：**不再是抄默认值——每个池都显式拧了旋钮**。

```
import requests
from user_agent_pool import UserAgentPool, UAStrategy
from dns_resolver_pool import DNSResolverPool, SelectStrategy
from nurture_pool import PoolOrchestrator

# ── UA 池：均匀轮换替代默认加权 ──
ua = UserAgentPool(strategy=UAStrategy.UNIFORM)
# → UserAgentPool(desktop=161, mobile=676, tablet=17)

# ── DNS 池：轮询 + 仅国内 7 台 ──
dns = DNSResolverPool(
    strategy=SelectStrategy.ROUND_ROBIN,
    regions=("domestic",),               # 114.114.114.114 / 223.5.5.5 等 7 台
)

# ── 编排器（不用短别名，直接用完整 API）──
orch = PoolOrchestrator(ua=ua, dns=dns)

with dns:                                # DNSResolverPool 也支持 with
    c = orch.next()
    resp = requests.get("https://www.baidu.com",
                        headers=c.ua, timeout=10)
print(resp.status_code)                  # → 200
```

**逐行讲解**

1. `import` 这里导入了 `UAStrategy` 和 `SelectStrategy`——开箱即用不需要你导入策略枚举（因为全用默认值），但你显式拧旋钮就必须导入。

2. `UserAgentPool(strategy=UAStrategy.UNIFORM)` ——这行做了两件事：
   - 创建了 UA 池（和开箱即用一样会加载 854 条内置种子）
   - 把选取策略从默认的"权重优先"改为"均等随机"
   - 池初始化时 `_init_defaults()` 加载 `ua_seeds.json`，耗时约 30~50ms，但只在创建时执行一次

3. `DNSResolverPool(strategy=SelectStrategy.ROUND_ROBIN, regions=("domestic",))` ——三个决定：
   - 策略不再用延迟加权，改用轮询：第 1 次解析问 114、第 2 次问 223、第 3 次问 119……14 台轮一遍后从头开始
   - `regions=("domestic",)` 只启用国内 7 台 DNS。注意末尾逗号不能少——`("domestic")` 会被 Python 当成字符串而不是元组，日志里会看到 "已加载 14 台 DNS" 而不是 7 台
   - 其他旋钮保持默认：缓存 300 秒、连续失败 3 次隔离、隔离 120 秒复活

4. `PoolOrchestrator(ua=ua, dns=dns)` ——用完整的编排器类替代开箱即用的短别名 `nurture_pool.combo()`。两者做的事一样——把 UA 池和 DNS 池绑在一起。但显式创建让你看到背后是什么。

5. `with dns:` ——DNS 池的上下文管理器，做了两件事：
   - `__enter__`：把 `socket.getaddrinfo` 替换为池内的自定义解析函数（monkey-patch）
   - `__exit__`：恢复原始 `socket.getaddrinfo`
   - 这样 `requests.get()` 内部的 DNS 解析就走池内 7 台 DNS 了

6. `orch.next()` ——编排器一次性调用所有池的选资源方法：
   - 遍历注册表 → `isinstance(ua, UserAgentPool)` 命中 → 调 `ua.get_headers("desktop")` → 拿到 14 项请求头
   - 遍历注册表 → `isinstance(dns, DNSResolverPool)` 命中 → 调 `dns.get_server()` → 拿到当前 DNS 服务器 IP
   - 返回 `PoolCombo`，包含 `c.ua` 和 `c.dns`

7. `requests.get(headers=c.ua)` ——`c.ua` 是完整的请求头字典（User-Agent + Accept + Accept-Language + Sec-Ch-Ua + ……共 14 项）。**不需要**你再手动拼 `{"User-Agent": "..."}`。

## 多线程

和开箱即用的区别：**显式创建编排器 + 加载自己的数据源**。

```
import requests
from concurrent.futures import ThreadPoolExecutor, as_completed
from user_agent_pool import UserAgentPool, UAStrategy
from dns_resolver_pool import DNSResolverPool, SelectStrategy
from nurture_pool import PoolOrchestrator

# ── UA 池：加载自己的 UA 数据 ──
ua = UserAgentPool(strategy=UAStrategy.UNIFORM)
# 如果你有积累的 UA 列表，加载进来
# ua.load_from_file("my_ua.json")   # JSON: [{"ua": "...", "category": "desktop", "weight": 5}]
# ua.load_from_file("my_ua.csv")    # CSV:  ua,category,weight

dns = DNSResolverPool(strategy=SelectStrategy.ROUND_ROBIN, regions=("domestic",))
orch = PoolOrchestrator(ua=ua, dns=dns)

with dns:
    def fetch(i):
        c = orch.next()
        resp = requests.get("https://www.baidu.com",
                            headers=c.ua, timeout=10)
        return i, resp.status_code

    with ThreadPoolExecutor(max_workers=10) as ex:
        for f in as_completed([ex.submit(fetch, i) for i in range(10)]):
            i, status = f.result()
            print(f"#{i}: {status}")   # → 10 个 200，均匀轮换
```

**逐行讲解**

1. `ua.load_from_file("my_ua.json")` ——这是数据源旋钮。支持三种格式：
   - **JSON**：`[{"ua": "Mozilla/5.0 ...", "category": "desktop", "weight": 5}, ...]` ——每条可以单独设权重。`category` 必须是 `"desktop"`、`"mobile"` 或 `"tablet"`。
   - **CSV**：第一行 `ua,category,weight`，后续每行一条。最简单。
   - **JSONL**：每行一个 JSON 对象，支持完整 Header Profile（和 `headers_pool.jsonl` 格式一样）。
   - 如果不传 `category`，池会自动根据 UA 字符串推测分类（检测 `mobile`/`iphone`/`android`/`tablet` 等关键词）。

2. 线程安全不用你管——UA 池的 `ReadWriteLock` 让 10 个线程同时 `orch.next()` 可以并发进入（读锁共享）。DNS 池的 `threading.Lock` 保护服务器状态修改（写操作），但取值（读操作）极快，不会成为瓶颈。

3. `with dns:` 在 ThreadPoolExecutor 外层——这意味着 10 个线程共享同一个 socket patch。`requests.get()` 在每个线程内、socket 层会走同一次 patch——不需要每线程单独 `with dns:`。

## 多进程

和开箱即用的区别：**每子进程加载自己的数据 + 关锁省开销**。

```
import requests
from concurrent.futures import ProcessPoolExecutor
from user_agent_pool import UserAgentPool, UAStrategy
from dns_resolver_pool import DNSResolverPool, SelectStrategy
from nurture_pool import PoolOrchestrator

def fetch(i):
    # ── 每子进程独立建池：加载自己的数据 + 关锁 ──
    ua = UserAgentPool(strategy=UAStrategy.UNIFORM, thread_safe=False)
    # ua.load_from_file("my_ua.json")   # 每子进程独立加载
    dns = DNSResolverPool(
        strategy=SelectStrategy.ROUND_ROBIN,
        regions=("domestic",),
        thread_safe=False,              # 子进程内单线程，关锁
    )
    orch = PoolOrchestrator(ua=ua, dns=dns)

    with dns:
        c = orch.next()
        resp = requests.get("https://www.baidu.com",
                            headers=c.ua, timeout=10)
    return i, resp.status_code

if __name__ == '__main__':
    with ProcessPoolExecutor(max_workers=4) as ex:
        for i, status in [f.result() for f in ex.map(fetch, range(8))]:
            print(f"#{i}: {status}")   # → 8 个 200
```

**逐行讲解**

1. `thread_safe=False` ——子进程内只有一个线程（你），所有锁变成空操作。节省了获取/释放锁的 CPU 开销。

2. 每个子进程独立建池——`UserAgentPool()` 和 `DNSResolverPool()` 在 `fetch()` 函数内创建，这意味着：
   - 每个子进程有自己的一份数据和锁
   - 子进程之间完全不共享状态（符合多进程的内存隔离模型）
   - 如果你想每子进程用不同的 UA 数据集，加载不同文件即可

3. `if __name__ == '__main__':` ——Windows 下必须有，否则子进程会递归创建子进程。

4. `with dns:` 在子进程内执行——socket patch 是进程级别的，子进程 A patch 不影响子进程 B，也不影响主进程。

## 异步

和开箱即用的区别：**显式创建异步编排器 + 代理池自动维护**。

```
import asyncio, aiohttp
from user_agent_pool.pool_async import AsyncUserAgentPool
from dns_resolver_pool.pool_async import AsyncDNSResolverPool
from proxy_pool.pool_async import AsyncProxyPool
from nurture_pool.orchestrator_async import AsyncPoolOrchestrator

ua = AsyncUserAgentPool()              # 异步 UA 池（类名前加 Async）
dns = AsyncDNSResolverPool()           # 异步 DNS 池

# ── 代理池：在 main() 里 await 初始化 ──
proxy = None  # 占位

async def fetch(session, i):
    c = await orch.next()
    async with session.get("https://www.baidu.com",
                           headers=c.ua) as resp:
        return i, resp.status

async def main():
    global proxy, orch
    proxy = AsyncProxyPool(
        min_alive=5,
        # auto_refill_url="http://your-provider.com/api?key=xxx",
    )
    await proxy.add_proxy({"scheme": "http", "host": "1.2.3.4", "port": 8080})
    orch = AsyncPoolOrchestrator(ua=ua, dns=dns, proxy=proxy)

    connector = aiohttp.TCPConnector(resolver=dns.create_resolver())
    async with aiohttp.ClientSession(connector=connector) as session:
        for i, status in await asyncio.gather(
            *[fetch(session, i) for i in range(10)]
        ):
            print(f"#{i}: {status}")   # → 10 个 200

asyncio.run(main())
```

**逐行讲解**

1. **异步池的导入路径**——和同步版不在同一个模块：
   - `from user_agent_pool.pool_async import AsyncUserAgentPool`（不是 `from user_agent_pool import ...`）
   - `from dns_resolver_pool.pool_async import AsyncDNSResolverPool`
   - `from proxy_pool.pool_async import AsyncProxyPool`
   - `from nurture_pool.orchestrator_async import AsyncPoolOrchestrator`

2. **`AsyncProxyPool(min_alive=5)`** ——设置了最低存活代理数为 5。配置完成后：
   - 调用 `proxy.auto_maintain()` 时：评分淘汰低分代理 → 统计存活数 → 如果 < 5 → 自动从 `auto_refill_url` 拉新代理
   - 生产环境把 `auto_maintain()` 放进定时任务（每 5 分钟一次），代理池自己管自己
   - `auto_maintain()` 有 60 秒冷却期——连续调用只有第一次生效

3. **`await proxy.add_proxy(...)`** ——`AsyncProxyPool` 的方法是 `async` 的，必须 `await`。不能在模块级（`async def` 外面）调用，所以整个初始化放在 `main()` 里。

4. **`dns.create_resolver()`** ——返回一个给 aiohttp 用的异步 DNS resolver。aiohttp 内部要用 `TCPConnector(resolver=...)` 接上。这样 aiohttp 的所有 DNS 解析都走池内 14 台轮换。

5. **`await orch.next()`** ——异步编排器的 `next()` 也是 `async` 的，需要 `await`。内部逻辑和同步版完全一样。

## Scrapy 中间件

和开箱即用的区别：**显式配置每个池的策略 + 代理自动维护**。

### 完整项目骨架

你需要三个文件（放到同一个 Scrapy 项目目录下）：

**① `middlewares.py`**

```
# middlewares.py —— UA + DNS + Proxy 三件套，全显式配置
from user_agent_pool import UserAgentPool, UAStrategy
from dns_resolver_pool import DNSResolverPool, SelectStrategy
from proxy_pool import ProxyPool
from nurture_pool import PoolOrchestrator

class ResourcePoolMiddleware:
    def __init__(self):
        self.ua = UserAgentPool(strategy=UAStrategy.UNIFORM)
        self.dns = DNSResolverPool(
            strategy=SelectStrategy.ROUND_ROBIN,
            regions=("domestic",),
        )
        self.proxy = ProxyPool(min_alive=5)
        # 如果你有代理供应商 API：
        # self.proxy.load_from_url("http://provider.com/api?key=xxx")

        self.orch = PoolOrchestrator(ua=self.ua, dns=self.dns, proxy=self.proxy)

    def process_request(self, request, spider):
        self.dns.__enter__()              # patch socket
        c = self.orch.next()
        request.headers.update(c.ua)
        if c.proxy:
            request.meta["proxy"] = c.proxy.get("http")

    def process_response(self, request, response, spider):
        self.dns.__exit__(None, None, None)
        return response

    def process_exception(self, request, exception, spider):
        self.dns.__exit__(None, None, None)
```

**② `settings.py`（在已有 settings 里添加）**

```python
DOWNLOADER_MIDDLEWARES = {
    "myproject.middlewares.ResourcePoolMiddleware": 543,
}
```

数字 `543` 是中间件优先级。Scrapy 内置中间件在 400~600 之间，543 在中间偏后——确保 UA 头不会被后面的中间件覆盖。

**③ `spider.py`（示例）**

```
import scrapy

class BaiduSpider(scrapy.Spider):
    name = "baidu"
    start_urls = ["https://www.baidu.com"]

    def parse(self, response):
        yield {"url": response.url, "status": response.status}
```

然后在项目根目录运行：

```bash
scrapy crawl baidu -O output.json
# → 写入 output.json
```

**逐行讲解**

1. **`__init__`** ——中间件实例化时一次性创建所有池。Scrapy 框架保证只会创建一次（单例模式），之后所有请求共享同一个中间件实例。

2. **`process_request`** ——每次请求前执行：
   - `self.dns.__enter__()`：patch socket，让 Scrapy Downloader 内部的 DNS 解析走池内 7 台
   - `self.orch.next()`：获取当次请求的 UA + DNS + Proxy
   - `request.headers.update(c.ua)`：把 14 项完整请求头写到 Scrapy Request 上
   - `request.meta["proxy"]`：Scrapy 识别这个 meta key 会用代理发包

3. **`process_response`** ——每次响应后执行：`dns.__exit__()` 恢复 socket。正常响应和异常响应都会走到这里（Scrapy 引擎链保证）。

4. **`process_exception`** ——请求抛异常时执行：同样恢复 socket。**非常重要**——如果少了这步，socket patch 会残留，后续所有请求的 DNS 解析都可能异常。

## 养成定制

开箱即用里你会 `feed_*()` 了。现在你需要知道：怎么批量导入/导出、怎么按批次管理、怎么跨项目共享。

### 批量导入（`import_*`）

`import_ua()` / `import_proxy()` / `import_dns()` 接受符合 schema 的列表，自动去重后写入。

```python
import nurture_pool

# ── 批量导入代理 ──
result = nurture_pool.import_proxy([
    {"proxy": "1.2.3.4:8080", "weight": 5},
    {"proxy": "5.6.7.8:3128", "weight": 8, "region": "china"},
    # 标准格式见 nurture_pool/data/schema/proxy_format.json
])
# → {"added": 2, "skipped": 0, "errors": 0}

# ── 批量导入 DNS ──
result = nurture_pool.import_dns([
    {"ip": "172.16.0.1", "name": "内网 DNS", "region": "domestic", "weight": 10},
])
# → {"added": 1, "skipped": 0, "errors": 0}
```

### 养成的 5 种写法

养成 API 和池操作一样支持 5 种写法——同步/异步/单线程/多线程/Scrapy 均可调用。

```
# 单线程
nurture_pool.feed_ua("Mozilla/5.0 ...", weight=8)

# 异步（feed_* 是同步的，可在 async 函数内直接调用，底层文件写入是原子的）
async def main():
    nurture_pool.feed_proxy("1.2.3.4:8080:user:pass")
    print(nurture_pool.status())

# Scrapy 爬虫内喂养
class MySpider(scrapy.Spider):
    def parse(self, response):
        # 喂入从响应里提取的好代理
        nurture_pool.feed_proxy("5.6.7.8:8080", weight=7)
```

### 按批次管理

每条养成数据自带 `batch` 批次号（默认自动生成时间戳）。`export_fed()` 可按批次过滤。

```
# 导出所有养成数据
nurture_pool.export_fed("proxy", "./backup/")

# 导出指定批次
nurture_pool.export_fed("proxy", "./jd_project/", batch="20260527_001")

# 查看某批次的条目
fed_items = nurture_pool.list_fed("proxy")
for item in fed_items:
    print(f"{item['host']}:{item['port']}  batch={item.get('batch')}")

# 清除养成（仅删 fed 条目，内置数据不受影响）
nurture_pool.reset("proxy")  # 清空代理养成
nurture_pool.reset()          # 清空全部养成
```

> **这发生了什么**
>
> `import_*()` 是 `feed_*()` 的批量版——共享同一套去重/权重更新/自动分类逻辑。`batch` 批次号让你能按项目/按日期分组管理养成数据，跨项目导出/导入。
>
> 标准格式定义在 `nurture_pool/data/schema/` 目录，三种格式（`ua_format.json` / `dns_format.json` / `proxy_format.json`）都是完整的 JSON Schema，可直接用于数据校验。

## 跑不通？看这里

| 现象 | 原因 | 解法 |
|------|------|------|
| 导入了策略但没效果 | 忘记传给构造函数 | `UserAgentPool(strategy=UAStrategy.UNIFORM)`，不是 `ua.strategy = ...` 事后赋值 |
| `load_from_file` 返回 0 | 文件路径不对或格式不匹配 | 确认文件存在；JSON 格式是 `[{...}]` 数组；CSV 第一行是 `ua,category,weight` |
| `regions=("domestic",)` 报错 | 拼写错误 | 只有 `"domestic"` 和 `"overseas"` 两个选项 |
| `regions` 没生效——仍是 14 台 | 少写了末尾逗号 | `("domestic",)` 不是 `("domestic")`——后者会被当成字符串，不是元组 |
| 代理自动补充不生效 | `auto_refill_url` 没设或网络不通 | 先手动 `load_from_url()` 测试 URL 可用，再开 `auto_maintain()` |
| 异步版 `add_proxy` 报 RuntimeWarning | 在模块级没用 `await` | `AsyncProxyPool` 的方法是 async 的，必须在 `async def` 里 `await` |
| Scrapy 中间件请求后 socket 异常 | 漏了 `process_exception` 的 `dns.__exit__` | 补上 `process_exception` 里的 restore |
