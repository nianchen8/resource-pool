# 开箱即用

> 百度首页 `https://www.baidu.com` 为目标。三件套一把抓：UA 请求头 + DNS 池加速 + 代理。每段复制就能跑。

## 先看这个：你的爬虫为什么一跑就封？

新手最常见的误区：**只换 User-Agent**。

真实浏览器发请求时至少带 5 个关键请求头（Accept / Accept-Language / Sec-Fetch-* 等），而 `requests` 默认只发一个 `User-Agent`。服务器一看——"缺了 N 个头，100% 是脚本"——秒封。

**nurture-pool 做的事**：UA 池给你补全 14 个字段的完整请求头，DNS 池让域名解析走 14 台 DNS 轮换（避免单台 DNS 被限流），Proxy 池管代理的评分/淘汰/补充。三件套协同——每次请求都是一套全新的浏览器指纹。

下面 5 种写法都经过实测验证。目标：`https://www.baidu.com`，返回 200。

---

## 单线程

最简入门——3 步：拿请求头 → DNS 加速 → 编排器一把抓。

```python
import nurture_pool, requests
from user_agent_pool import UserAgentPool

# ── 创建三件套 ──
ua_pool = UserAgentPool()              # 自动加载 854 种子，可重组 31,496 个 UA
# → UserAgentPool(desktop=161, mobile=676, tablet=17)
dns = nurture_pool.DNS()              # 惰性包装器，首次使用时自动初始化
proxy = nurture_pool.Proxy("1.2.3.4:8080")  # 直接填地址；没有代理就去掉这行

# ── ① 获取完整请求头 ──
headers = ua_pool.get_headers("desktop")
# → User-Agent / Accept / Accept-Language / Sec-Ch-Ua / Sec-Fetch-* 等 14 字段

# ── ② DNS 池加速 —— with 块内 socket 层自动 patch，14 台 DNS 轮换 ──
with dns:                              # 进入 → patch socket.getaddrinfo
    resp = requests.get("https://www.baidu.com",
                        headers=headers, timeout=10)
print(resp.status_code)                # → 200

# ── ③ 编排器一把抓（UA + DNS + Proxy 一键组合）──
with dns:
    c = nurture_pool.combo(ua=ua_pool, dns=dns, proxy=proxy)
    resp = requests.get("https://www.baidu.com",
                        headers=c.ua, proxies=c.proxy, timeout=10)
# → c.ua     = 14 字段请求头 dict
#    c.proxy  = {"http": "http://1.2.3.4:8080", "https": "..."}
```

> **这发生了什么**
>
> ① `get_headers("desktop")` 不是随便拼——UA 池从 854 条种子里拆解 OS/版本/引擎零件，跨零件随机重组出独立 UA，再按引擎家族（Chromium/Firefox/Safari）组装配套的 Accept、Sec-Ch-Ua、Sec-Fetch-* 等字段。**同一套 headers 里的所有字段来自同一个设备指纹**，不会出现 Chrome 的 Accept 配 Firefox 的 Accept-Language 这种低级错误。
>
> ② `with dns:` 进入时悄悄替换了 Python 底层的 `socket.getaddrinfo`，之后 `requests` 发出的所有 DNS 解析自动走池内 14 台服务器轮换，单台挂了自动跳过。离开 `with` 块自动恢复。
>
> ③ `combo()` 把 UA 池、DNS 池、Proxy 池串成一个 `PoolCombo` 对象——`c.ua` 拿请求头，`c.proxy` 拿代理配置，一行搞定。

> **没有代理？** 把 `proxy = nurture_pool.Proxy(...)` 那行删掉，`combo()` 里也去掉 `proxy=proxy`，`proxies=c.proxy` 也删掉。代码照样跑。

## 多线程

UA 池内置 `ReadWriteLock`，多线程读自动并发安全——不需要你手动加锁。

```python
import nurture_pool, requests
from concurrent.futures import ThreadPoolExecutor, as_completed
from user_agent_pool import UserAgentPool

# ── 三件套（UA 池内置 ReadWriteLock，多线程读自动并发安全）──
ua_pool = UserAgentPool()              # → UserAgentPool(desktop=161, mobile=676, tablet=17)
dns = nurture_pool.DNS()              # → DNS(not loaded)，惰性初始化
proxy = nurture_pool.Proxy("1.2.3.4:8080")  # → Proxy(1 queued)

# ── with dns 包住整个线程池，所有线程的 DNS 解析走池内轮换 ──
with dns:
    def fetch(i):
        c = nurture_pool.combo(ua=ua_pool, dns=dns, proxy=proxy)
        resp = requests.get("https://www.baidu.com",
                            headers=c.ua, proxies=c.proxy, timeout=10)
        return i, resp.status_code

    with ThreadPoolExecutor(max_workers=10) as ex:
        for f in as_completed([ex.submit(fetch, i) for i in range(10)]):
            i, status = f.result()
            print(f"#{i}: {status}")   # → 10 个 200，每次 UA/代理 不同
```

> **这发生了什么**
>
> `with dns:` 包在整个线程池外面——patched 的 `socket.getaddrinfo` 是进程级全局的，所有线程共享。`UserAgentPool` 内置 `ReadWriteLock`，10 个线程同时 `get_headers()` 可以并发进入（读锁共享），不会排队等锁。
>
> 每次 `fetch()` 里调 `combo()` 都会拿到一套不同的 UA + 代理，10 次请求指纹全不同。

## 多进程

多进程不能共享内存对象——每个进程独立建池。

```python
import nurture_pool, requests
from concurrent.futures import ProcessPoolExecutor
from user_agent_pool import UserAgentPool

# ── 每个进程独立建池，thread_safe=False 消除锁开销 ──
def fetch(i):
    ua_pool = UserAgentPool(thread_safe=False)  # 子进程内独立加载
    dns = nurture_pool.DNS()
    proxy = nurture_pool.Proxy("1.2.3.4:8080")

    with dns:                          # 子进程内独立 patch socket
        c = nurture_pool.combo(ua=ua_pool, dns=dns, proxy=proxy)
        resp = requests.get("https://www.baidu.com",
                            headers=c.ua, proxies=c.proxy, timeout=10)
    return i, resp.status_code

if __name__ == '__main__':
    with ProcessPoolExecutor(max_workers=4) as ex:
        for i, status in [f.result() for f in ex.map(fetch, range(8))]:
            print(f"#{i}: {status}")   # → 8 个 200
```

> **这发生了什么**
>
> Windows 下 `ProcessPoolExecutor` 用 `spawn()` 创建子进程（不是 Linux 的 `fork()`），每个子进程会重新导入当前模块。`if __name__ == '__main__':` 防止子进程递归创建进程池——**Windows 下没这行直接报 `RuntimeError`**。
>
> `thread_safe=False`：子进程内只有你一个线程在跑，关掉所有锁，零开销。
>
> 每个子进程独立 `UserAgentPool()`：自动加载 854 条种子、建立自己的 UA 零件池，互不干扰。

## 异步

同步版和异步版 API 完全对等——类名前面加 `Async`。

```python
import asyncio, aiohttp
from user_agent_pool.pool_async import AsyncUserAgentPool
from dns_resolver_pool.pool_async import AsyncDNSResolverPool
from nurture_pool.orchestrator_async import AsyncPoolOrchestrator

# ── 异步三件套 ──
ua_pool = AsyncUserAgentPool()         # → AsyncUserAgentPool(desktop=161, mobile=676, tablet=17)
dns = AsyncDNSResolverPool()           # → AsyncDNSResolverPool(alive=14/14)
orch = AsyncPoolOrchestrator(ua=ua_pool, dns=dns)  # 编排器

async def fetch(session, i):
    c = await orch.next()
    async with session.get("https://www.baidu.com",
                           headers=c.ua) as resp:
        return i, resp.status

async def main():
    # create_resolver() 让 aiohttp 的 DNS 解析走池内 14 台服务器
    connector = aiohttp.TCPConnector(resolver=dns.create_resolver())
    async with aiohttp.ClientSession(connector=connector) as session:
        for i, status in await asyncio.gather(
            *[fetch(session, i) for i in range(10)]
        ):
            print(f"#{i}: {status}")   # → 10 个 200

asyncio.run(main())
```

> **这发生了什么**
>
> `dns.create_resolver()` 返回一个对象，塞进 `aiohttp.TCPConnector(resolver=...)` 后，aiohttp 的 DNS 解析自动走池内 14 台 DNS 轮换——跟同步版的 `with dns:` 效果一样，只是异步版的正确姿势。
>
> `orch` 在模块级定义——`fetch()` 和 `main()` 都可直接引用，无需在 `main()` 里赋值。`asyncio.gather()` 让 10 个协程同时起飞。

## Scrapy 中间件

`middlewares.py`（放到你的 Scrapy 项目里）：

```python
# middlewares.py —— UA + DNS + Proxy 三件套
import nurture_pool
from user_agent_pool import UserAgentPool

class ResourcePoolMiddleware:
    def __init__(self):
        self.ua = UserAgentPool()
        self.dns = nurture_pool.DNS()
        # self.proxy = nurture_pool.Proxy("1.2.3.4:8080")  # 可选

    def process_request(self, request, spider):
        self.dns.__enter__()           # patch socket，DNS 走池内轮换
        request.headers.update(self.ua.get_headers())
        # request.meta["proxy"] = self.proxy.pick()          # 可选

    def process_response(self, request, response, spider):
        self.dns.__exit__(None, None, None)  # 请求完成后 unpatch
        return response
```

`settings.py` 启用：

```python
# settings.py
DOWNLOADER_MIDDLEWARES = {
    "myproject.middlewares.ResourcePoolMiddleware": 543,
}
```

```bash
# 运行：scrapy crawl baidu
$ scrapy crawl baidu -O output.json
# → 200 OK，title="百度一下，你就知道"
```

> **这发生了什么**
>
> Scrapy 发请求前走 `process_request`：patch socket DNS → 换 UA 请求头。请求回来后走 `process_response`：unpatch 恢复。
>
> `dns.__enter__()` / `dns.__exit__()` 就是同步版 `with dns:` 的手动版——因为 Scrapy 的 middleware 不走上下文管理器。**一定要在 `process_response` 里 unpatch**，否则之后所有请求的 DNS 解析都被 patch 了（看起来没事，但 socket 被永久替换不干净）。

## 养成：让池子越用越肥

遇到好用的资源别浪费——一条命令永久喂入池子，下次启动自动加载。

```python
import nurture_pool

# ── 喂一条新 UA（自动解析 browser/os/version，分类到 desktop/mobile/tablet）──
nurture_pool.feed_ua(
    "Mozilla/5.0 (Windows NT 10.0; Win64; x64) AppleWebKit/537.36 "
    "(KHTML, like Gecko) Chrome/148.0.0.0 Safari/537.36",
    weight=8,                          # 权重 1-10，越高越容易被选到
    profile="chrome_148_win",          # Header Profile 键（可选）
)
# → feed_ua: 已喂入 Mozilla/5.0 (Windows NT 10.0...) → 分类 'desktop', batch=20260527_143052

# ── 喂一个代理（支持 ip:port / ip:port:user:pass / http://ip:port）──
nurture_pool.feed_proxy("1.2.3.4:8080:user:pass", weight=8)
# → feed_proxy: 已喂入 http://1.2.3.4:8080, batch=20260527_143052

# ── 喂一台 DNS 服务器 ──
nurture_pool.feed_dns("8.8.8.8", name="Google DNS", region="overseas")
# → feed_dns: 已喂入 8.8.8.8 (Google DNS), batch=20260527_143052

# ── 查看养成统计 ──
print(nurture_pool.status())
# → {"ua": {"builtin": 854, "fed": 1, "total": 855}, "dns": {...}, "proxy": {...}}

# ── 验证养成代理质量（三次测试，失败自动导出列表）──
result = nurture_pool.validate_fed_proxies(max_retries=3, timeout=5)
print(f"通过 {result['passed']}, 失败 {result['failed']}")
# → 通过 5, 失败 1（失败列表 → proxy_failed_export.json）
```

> **这发生了什么**
>
> `feed_ua()` 把新 UA 写入 `user_agent_pool/ua_seeds.json`（与内置种子同一文件），标记 `source="fed"` + 当前批次号。`feed_proxy()` / `feed_dns()` 写入 `nurture_pool/data/` 下的持久化文件。**下次 `UserAgentPool()` / `ProxyPool()` 初始化时自动加载**。
>
> 批量导入用 `import_ua()` / `import_proxy()` / `import_dns()`，支持标准 JSON 格式（见 `nurture_pool/data/schema/`）。
>
> 养成数据在 pip upgrade 时会丢失——定期 `nurture_pool.export_fed("proxy", "./backup/")` 备份到安全目录。`nurture_pool.reset("proxy")` 一键清除养成数据。
>
> 养成代理可用 `validate_fed_proxies()` 批量验证连通性（三次测试），失败的代理自动导出到 `proxy_failed_export.json` 方便排查。单代理可用 `probe_proxy()` 探测。

## 跑不通？看这里

| 现象 | 原因 | 解法 |
|------|------|------|
| `DNS(not loaded)` | DNS 池惰性初始化，首次使用才加载 | 正常，进 `with dns:` 或调 `dns.resolve()` 后自动完成 |
| 多进程报 `RuntimeError` | Windows 下缺 `if __name__ == '__main__':` | 加上（见上面多进程示例） |
| `create_resolver()` 报 `AttributeError` | 旧版本 bug，返回裸函数 | `pip install -U nurture-pool` |
| `PoolExhaustedException` | 池内资源全部耗尽 | 检查网络、代理是否过期、DNS 是否全挂 |
| 状态码不是 200 | 百度反爬升级 / 代理被封 | 换目标 URL 试试；单池测试定位是 UA 还是代理的问题 |
| `feed_*()` 写入后 pool 没变化 | 数据文件需重新创建池才会加载养成条目 | 养成数据是持久的——下次 `UserAgentPool()` / `ProxyPool()` 初始化时自动读取 |
| `export_fed()` 报路径不存在 | 还没有养成数据 | 先用 `feed_*()` 喂几条，`status()` 确认有数据后再导出 |
